# 客服 FAQ 自动分类 - 代码审查与改进

## 一、Code Review：发现的问题（按严重程度排序）

### 问题 1 [严重] API Key 硬编码在源码中
- **位置**: `classifier.py` 第 11 行
- **原代码**: `openai.api_key = "sk-proj-abc123..."`
- **影响**: 密钥泄露到版本控制系统，可能导致未授权调用、费用损失，属于严重安全漏洞
- **修复**: 改为从环境变量 `OPENAI_API_KEY` 读取，提供 `.env.example` 模板

### 问题 2 [严重] 无任何错误处理与重试机制
- **位置**: `classify_question()` 和 `batch_classify()` 函数
- **影响**: 网络波动、API 限流或服务端错误将直接导致脚本崩溃，中断批量任务，且无法自动恢复
- **具体崩溃点**:
  - `response.choices[0].message.content` 未检查 `choices` 是否为空、`content` 是否为 `None`（内容过滤时返回 None）
  - `batch_classify` 中途崩溃时已处理数据全部丢失
  - 输入文件格式错误（非法 JSON、缺少字段）直接抛异常
  - 单条数据处理异常导致整个批处理中断
- **修复**: 添加 try/except + 指数退避重试机制（最多 3 次），API 返回值逐层校验，单条失败不影响整体，批处理崩溃时保存 `.partial` 文件（详见改进 8）

### 问题 3 [高] Prompt 设计简陋，分类准确率低
- **位置**: `classify_question()` 中的 prompt
- **问题**:
  - 无 system prompt，所有指令塞在 user message 中
  - 类别仅列出名称，无定义和示例
  - 无边界情况指引（多类别、退款进度等）
  - 输出格式为纯文本，难以程序化验证
  - 无 Few-shot 示例引导模型行为
- **影响**: 模型对类别边界理解不清，导致误分类（如退款进度误判为物流查询、投诉建议误判为退款退货等）
- **修复**: 重新设计 prompt（详见第二节）

### 问题 4 [高] 无输出验证
- **位置**: `classify_question()` 第 32 行
- **原代码**: `result = response.choices[0].message.content.strip()` 直接返回
- **影响**: LLM 可能返回非标准类别名（如"退换货"而非"退款退货"），导致下游系统报错
- **修复**: 添加 `parse_category()` 函数，支持 JSON 和纯文本解析，并校验是否为合法类别

### 问题 5 [中] 使用已弃用的 OpenAI API 风格
- **位置**: `openai.api_key = ...` 和 `openai.chat.completions.create(...)`
- **影响**: 这是 openai v0.x 的写法，v1.x+ 已不兼容，可能直接报错
- **修复**: 改为 `client = OpenAI()` + `client.chat.completions.create()` 的 v1.x 风格

### 问题 6 [中] 串行调用 API，效率低下
- **影响**: 对每条问题单独发起一次 HTTP 请求，耗时随数据量线性增长，无法利用并发提升吞吐量
- **修复**: 添加 `asyncio` + `Semaphore` 并发支持，`--async` 参数启用

### 问题 7 [低] 无日志记录
- **影响**: 生产环境难以排查问题
- **修复**: 使用 logging 模块记录关键信息

### 问题 8 [低] 缺乏结果校验与可测试性
- **影响**: 若模型返回非预期字符串，脚本会直接写入 JSON，无任何校验；同时无法离线测试
- **修复**: 添加 Mock 模式（基于预定义规则映射），用于单元测试和离线评估

## 二、Prompt 改进

### 改动对比

| 改动项 | 原版 v1.0 | 改进版 v2.0 | 理由 |
|--------|-----------|-------------|------|
| System Prompt | 无 | 有完整角色定义和指令 | 分离系统指令和用户输入，让模型更好理解角色和任务边界 |
| 类别定义 | 仅列出名称 | 包含定义 + 典型场景 | 帮助模型理解每个类别的语义边界，减少误分类 |
| 分类注意事项 | 无 | 7 条明确规则 | 解决边界模糊问题（退款进度归退款退货、多诉求取主诉求、投诉 vs 退款等） |
| Few-shot 示例 | 无 | 8 个精选示例 | 直接展示期望行为，尤其覆盖易错边界案例 |
| 输出格式 | "直接回复类别名称" | JSON 格式含 reasoning 字段 | 思维链提升准确率；结构化输出便于程序解析和验证 |
| 多类别处理 | 无指引 | "以主要诉求为准" | 解决用户问题涉及多个类别时的歧义 |
| 投诉 vs 退款 | 无指引 | 明确规则 + Few-shot | "退货流程太麻烦"归投诉建议而非退款退货 |

### Few-shot 示例设计思路

精选 8 个示例，覆盖以下关键边界场景：

| 示例 | 覆盖场景 | 设计理由 |
|------|---------|---------|
| "退款什么时候能到账" | 退款进度 vs 物流 | 原版易误判为物流查询 |
| "包裹显示已签收但我没收到" | 物流关键词扩展 | 原版缺少"包裹/签收"关键词 |
| "怎么修改绑定的手机号" | 账号关键词扩展 | 原版缺少"手机号"关键词 |
| "这个手机壳是硅胶的还是塑料的" | 商品咨询 | 原版关键词不足 |
| "你们这个退货流程也太麻烦了吧" | 投诉 vs 退款 | **最易错**：含退货关键词但核心是投诉 |
| "建议你们增加夜间配送选项" | 投诉建议 vs 物流 | 含配送关键词但核心是建议 |
| "我想问下这个退款的事顺便看看快递到没到" | 多诉求取主 | 退款为主，物流为辅 |
| "嗯嗯好的谢谢" | 其他/闲聊 | 无明确诉求 |

## 四、工程化改进

### 改进 1：API Key 安全管理
- 从环境变量 `OPENAI_API_KEY` 读取，不再硬编码
- 提供 `.env.example` 模板和 `.gitignore`
- 启动时检查，未设置则抛出明确错误

### 改进 2：错误处理 + 重试机制
- `classify_question()` 添加 try/except 包裹
- 指数退避重试（2^n 秒间隔，最多 3 次，可通过环境变量配置）
- 单条失败返回 `{"category": "其他", "error": "..."}` 而非崩溃

### 改进 3：输出验证
- `parse_category()` 函数支持 JSON 和纯文本两种格式解析
- 校验返回的类别是否在 `VALID_CATEGORIES` 列表中
- 无法解析时默认归入"其他"并记录 warning 日志

### 改进 4：异步并发支持
- 使用 `asyncio` + `Semaphore` 实现并发请求
- `--async` 参数启用，默认并发数 5
- 大幅缩短批量处理时间

### 改进 5：模块化代码结构
- `config.py` - 配置常量
- `client.py` - OpenAI 客户端管理
- `prompts.py` - Prompt 模板（含 Few-shot）
- `parser.py` - 输出解析与验证
- `mock.py` - Mock 模式（模拟原 prompt 误判 + 改进 prompt 行为）
- `classifier.py` - 核心分类逻辑

### 改进 6：单元测试
- 30 个测试用例覆盖：解析器、Mock 行为、Prompt 内容、端到端准确率、崩溃抵抗
- `pytest test_classifier.py -v` 运行

### 改进 7：评估框架
- 独立的 `evaluate.py` 评估脚本
- 支持 mock 模式和真实 API 模式
- 自动对比改进前后准确率，含各类别细分和逐条对比

### 改进 8：偶发崩溃全面修复

原始代码上线后"偶发报错"的根因是多种异常场景未被处理。改进版对以下 10 个崩溃场景全部做了防护：

| # | 崩溃场景 | 触发条件 | 修复方式 | 测试覆盖 |
|---|---------|---------|---------|---------|
| 1 | API 返回空 choices | 模型异常响应 | 检查 `response.choices` 是否为空，空则重试 | - |
| 2 | API 返回 None content | 内容过滤/function call 触发安全拦截 | 检查 `content is None`，None 则重试 | - |
| 3 | API 返回空字符串 | 模型返回空内容 | 检查 `raw` 是否为空，空则重试 | - |
| 4 | API 网络错误/限流 | 网络波动、429 限流、服务端 5xx | 指数退避重试（最多 3 次） | - |
| 5 | 批处理中途崩溃 | 处理到第 N 条时异常 | 单条 try/except 包裹，失败记入 error 不影响后续；异步模式崩溃时保存 `.partial` 文件 | `test_batch_classify_partial_save_on_item_error` |
| 6 | 输入文件不存在 | 文件路径错误 | 捕获 `FileNotFoundError` 并记录日志 | `test_batch_classify_missing_file` |
| 7 | 输入 JSON 格式错误 | 文件内容不是合法 JSON | 捕获 `JSONDecodeError`，抛出明确 `ValueError` | `test_batch_classify_invalid_json` |
| 8 | 输入不是列表 | JSON 是对象而非数组 | 检查 `isinstance(questions, list)` | `test_batch_classify_non_list_input` |
| 9 | 单条数据缺少 question 字段 | 数据格式不完整 | 检查 `"question" in item`，跳过无效条目并记录 warning | `test_batch_classify_skips_invalid_items` |
| 10 | LLM 返回非标准类别名 | 返回"退换货"等非预定义名称 | `parse_category()` 校验合法类别，不合法则归入"其他" | `TestParser` 系列 |

**核心设计原则**：单条失败不影响整体，任何异常都有兜底（归入"其他" + 记录 error），批处理不会因单条数据问题而中断。

## 五、如何运行代码

### 环境要求
- Python 3.8+
- 安装依赖：`pip install -r requirements.txt`

### Mock 模式（无需 API Key）
```bash
# 运行评估（对比改进前后）
python evaluate.py --mock

# 运行改进版分类器
python improved/classifier.py test_samples.json output.json --mock

# 运行单元测试
python -m pytest test_classifier.py -v
```

### 真实 API 模式
```bash
# 配置环境变量
copy .env.example .env
# 编辑 .env 填入 API Key

# 或直接设置环境变量
set OPENAI_API_KEY=sk-your-key-here

# 运行评估
python evaluate.py

# 运行改进版分类器（串行）
python improved/classifier.py test_samples.json output.json

# 运行改进版分类器（异步并发）
python improved/classifier.py test_samples.json output.json --async
```
